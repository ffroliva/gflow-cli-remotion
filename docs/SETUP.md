# Promo Pipeline — Operator Runbook

How to record a gflow demo tour and render the A/B promo matrix.

> Recording spends real Veo + Imagen credits. Always use a dedicated
> `promo-*` Chrome profile — never your main gflow profile.

---

## Prerequisites

| Tool | Min version | Notes |
|------|-------------|-------|
| Node | 20+ | `node --version` |
| pnpm | 11+ | `pnpm --version` |
| OBS Studio | 28+ | ships the WebSocket server (Tools → WebSocket Server Settings) |
| ffmpeg | 7+ | on PATH (`ffmpeg -version`) — confirmed working with 8.0.1 |
| gifski | 1.3+ | on PATH (`gifski --version`) — confirmed working with 1.34.0 |
| gflow-cli | latest | installed and on PATH (`gflow --version`) |

Install JS deps once: `pnpm install`.

---

## One-time setup

### 1. OBS WebSocket + scene

1. OBS → **Tools → WebSocket Server Settings** → enable, set a strong password.
2. Export that password to the shell you record from:
   - PowerShell: `$env:OBS_WS_PASSWORD = "…"`
   - bash: `export OBS_WS_PASSWORD="…"`
3. Import the committed scene collection: **Scene Collection → Import →**
   `docs/obs-scene.json`. It frames a 1920×1080 composition with a terminal
   pane on the left and a **Window Capture** of the headed Chrome window,
   with a Crop/Pad filter hiding the top-right account chip.

   > `docs/obs-scene.json` is exported by the operator (Phase 8.1) — it is
   > machine-specific (monitor layout, source names) and not generated by
   > this repo.

### 2. promo Chrome profile

Create a dedicated Chrome-strategy profile and log in to Google Flow:

```
gflow auth login --profile promo-XYZ --browser chrome
```

This writes the `.gflow_browser_strategy` marker the recorder checks. The
recorder **aborts** if the marker is missing or not `chrome`.

### 3. Hide the account chip (defence in depth)

Load `docs/promo.css` as a user-stylesheet on the promo profile (e.g. via
the Stylus extension). This hides the account chip even if the OBS crop
drifts.

### 4. Keep outputs out of git

The recorder writes to `~/gflow-output/promo/<run-id>/`. Add `gflow-output/`
to your global gitignore (`~/.gitignore_global`) so a stray run never lands
in a commit. (This repo already ignores `out/`.)

## Recording → rendering → GIF

```
# 1. Record the 4-phase tour (OBS must be running, password exported)
pnpm record-promo --profile promo-XYZ --run-id 2026-05-28-001

# 2. Render the matrix (6 hooks × {master, social} + one readme)
pnpm render-matrix --run-id 2026-05-28-001

# 3. Convert the README loop to GIF
pnpm post-gif --run-id 2026-05-28-001
```

Outputs:

| Stage | Location |
|-------|----------|
| recording | `~/gflow-output/promo/<run-id>/{run.json, master.mp4}` |
| renders | `./out/promo/<run-id>/{<hook>-master.mp4, <hook>-social.mp4, readme.mp4}` |
| GIF | `./out/promo/<run-id>/readme.gif` |

### Flags worth knowing

- `record-promo --dry-run` — runs the orchestrator against the `tests/fixtures/fake-gflow`
  stubs (no credits, FakeObsAdapter). Still requires a valid `promo-*` marker.
- `record-promo --force` — overwrite an existing `run.json` for the same run-id.
- `render-matrix --only PromoSocial` — render a single composition.
- `render-matrix --frames 30` — cap the frame range (cheap smoke).
- `GFLOW_PROFILE_ROOT=<dir>` — override the profile-root lookup (used by the
  integration tests; rarely needed by operators).

## 5-layer verification ledger

Never claim a recording succeeded on file-count alone. Capture evidence for
each layer in `docs/LIVE_VERIFICATION_promo.md` (created per release):

1. **File count** — `ls out/promo/<run-id>` shows the expected
   `(hooks × 2) + 1 readme [+ 1 gif]` files, and `run.json` + `master.mp4`
   exist in the recording dir.
2. **Magic bytes** — first bytes confirm real media, not empty stubs:
   - mp4: `xxd master.mp4 | head -1` shows an `ftyp` box.
   - gif: starts with `GIF89a`.
3. **Dimensions** — `ffprobe -v error -show_entries stream=width,height -of csv=p=0 <file>`
   reports 1920×1080 (master), 1080×1920 (social), 1280×720 (readme).
4. **Eye test** — play each format; confirm the hook reads correctly, the
   recording is sharp, and **the Google account chip is absent**.
5. **PII scan** — inspect the first frame + `run.json` for leaked
   email/token/home-path. `run.json` is already redacted by the orchestrator;
   confirm no `@`, no `C:\Users\<name>`, no tokens.

Only after all five pass is the run a keeper worth shipping as a Release asset.
